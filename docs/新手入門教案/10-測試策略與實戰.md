> 建議時數：120 min

---

## ① 為什麼先教這個？

不寫測試的程式碼是「可能會動」，有測試的程式碼是「確定會動」。測試是修改程式碼後不會造成回歸錯誤的安全網——這是從「學習專案」走向「生產專案」的關鍵一步。

## ③ 核心觀念

### 測試金字塔

```
        ∧
       /  \      e2e 測試（少且慢）
      /----\
     /      \    整合測試
    /--------\
   /          \  單元測試（多且快）
  --------------
```

### Jest 核心概念

| API | 用途 |
|-----|------|
| `jest.mock()` | 模擬模組 |
| `jest.fn()` | 建立 mock function |
| `mockResolvedValue()` | 定義 async mock 的回傳值 |
| `mockReturnValue()` | 定義 sync mock 的回傳值 |
| `expect().toHaveBeenCalledWith()` | 驗證呼叫參數 |

### @nestjs/testing

- 提供隔離的測試模組
- `Test.createTestingModule()` 建立測試環境

### e2e 測試

- 用 `supertest` 模擬完整 HTTP 請求
- 測試完整的請求 → 回應流程

### AAA 模式

```typescript
// Arrange（準備）
const mockBook = { id: 1, title: 'Test' };
jest.spyOn(service, 'findById').mockResolvedValue(mockBook);

// Act（執行）
const result = await controller.findOne(1);

// Assert（驗證）
expect(result).toEqual(mockBook);
```

## ④ 動手做

### 必做

1. **執行測試**
   - 執行 `npm run test`
   - 閱讀測試報告

2. **新增單元測試**
   - 在 `book.service.spec.ts` 新增一個測試方法：
   - `findById when not found throws exception`

### 延伸挑戰

1. 在 `book.api.controller.spec.ts` 新增測試：`update returns 200`
2. 執行 `npm run test:e2e` 觀察 e2e 測試的執行過程
3. 觀察測試速度差異：單元測試 vs e2e 測試

---

## ⑤ 踩坑提示

| 現象 | 原因 | 解法 |
|------|------|------|
| Mock 物件是 undefined | 忘記設定 mock 的回傳值 | 確保所有被呼叫的 mock 方法都有設定 `mockResolvedValue()` |
| e2e 測試連不上資料庫 | 測試環境沒有啟動 DB 容器 | e2e 測試前先 `docker compose up -d` |
